Robust to High-Concurrency Workloads

ThunderAgent maintains maximum achievable throughput even as the number of parallel workflows scales beyond GPU memory limits. Baseline systems suffer severe throughput collapse once the workload exceeds memory capacity, while ThunderAgent automatically adapts to the maximum available capacity with no need of manual tuning. This makes it reliable for real-world deployments where workload patterns are unpredictable.

4.2× Disk Memory Savings via Dynamic Resource Reclaim

ThunderAgent's lifecycle-aware garbage collector dynamically reclaims tool resources — such as Docker containers and temporary files — from completed workflows, preventing unused resources from accumulating and occupying disk space during long-running rollouts.

// Overview

ThunderAgent sits between agent clients and the infrastructure layer as an agentic workflow scheduler. It improves inference throughput of vLLM/SGLang across multiple GPU nodes through program-aware scheduling, and provides a unified tool management interface for resources like Docker containers and remote APIs.

Using ThunderAgent requires an LLM serving engine. Here we give an example of serving an agent with vLLM + Qwen3-32B.

Step 1 — Install vLLM & ThunderAgent

pip install vllm
git clone git@github.com:HaoKang-Timmy/ThunderAgent.git
cd ThunderAgent
pip install -e .
# Launch
thunderagent --backend-type vllm --backends http://localhost:8000 --port 9000

Step 4 — Send requests via OpenAI API to port 9000

# Python
from openai import OpenAI
client = OpenAI(base_url="http://localhost:9000/v1", api_key="EMPTY")
response = client.chat.completions.create(
model="Qwen/Qwen3-32B",
messages=[{"role": "user", "content": "Hello!"}],
extra_body={"program_id": "my_agent_001"}
)
print(response.choices[0].message.content)
// Agentic Inference Challenges

Agentic workflows consist of reasoning phases (which consume GPU resources) and acting phases (i.e., tool calling, which do not consume GPU). During the acting phase, request-aware serving engines (e.g., vLLM, SGLang) treat the KV cache of acting workflows as evictable idle memory. However, under high-concurrency serving scenarios, nearly all GPU memory is occupied by workflow KV caches (as shown in Step 1). Reasoning workflows then preempt the KV cache originally belonging to acting workflows (Step 2). When those acting workflows later re-enter the reasoning phase, they in turn evict KV caches of other workflows (Steps 3 & 4). This cascading eviction forces a large number of workflows to recompute the entire trajectory prefill from scratch, dramatically increasing the end-to-end latency of each request.

We evaluated serving mini-SWE-Agent with GLM4.6 on a single H100 node: as the number of parallel LLM agent workflows increases, the KV cache hit rate drops significantly, and the average end-to-end latency per request rises sharply.

In multi-node deployments, a standalone gateway does not have full workflow-level system state, and during KV cache thrashing it cannot reliably determine whether a workflow's KV cache has already been evicted. As a result, it cannot make a good tradeoff between KV-hit efficiency and workload balance. This often causes either memory hotspots or low KV hit rate, and only reaches suboptimal throughput.

We evaluated GLM-4.6 + mini-SWE-Agent on 2 H100 nodes with the SGLang gateway: memory imbalance emerges across nodes, reaching up to 51%.

Agentic workflows spawn tool environments (Docker containers, sandboxes, API sessions) that persist long after the workflow completes. Without lifecycle-aware reclamation, these zombie resources consume disk and memory, leading to 4× unnecessary storage overhead.

If tool resources are not reclaimed in time, disk pressure keeps accumulating until nodes hit storage bottlenecks.

// Our Method: Design Philosophy

Our method prioritizes program-aware scheduling: decouple scheduling from execution backends, preserve workflow continuity across pauses and resumes, and jointly optimize throughput, KV-hit efficiency, and resource lifecycle management.

  1. Use the program abstraction to define a pause/resume scheduler, and apply a time-decay factor to balance KV cahce hit rate and GPU utilization, so the system reaches optimal throughput.
  2. Treat new and paused programs as low cache-value traffic: new programs have little historical KV to reuse, and paused programs often have unstable or partially evicted KV after tool execution and waiting. Use these programs to improve cross-node load balance.
  3. Manage and reclaim tool resources from a program-level view.
